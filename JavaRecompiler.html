Why?

The idea behind a recompiler is to write code in Java that runs like native code written in C++.

I tried to port MFC to Java, and it didn't go well. The problems I had were all most likely related to Java/JNI/native code interaction.

  1. JVM executed some Java code starting with main().
  2. Somewhere along the way some window is created.
  3. Java code calls JNI CreateWindow() wrapper.
  4. CreateWindow() wrapper calls library CreateWindow() function.
  5. CreateWindow() sends some messages to the window. Window proc is implemented in native code.
  6. Native code uses JNI to invoke some Java method.
  7. Java method creates a child window through JNI CreateWindow() wrapper.
  8. CreateWindow() wrapper calls library CreateWindow() function.
  9. CreateWindow() sends some messages to the child window. Window proc is implemented in native code.
  10. Native code uses JNI to invoke some Java method.
  11. etc...

This JVM/JNI/Win32/JNI/JVM/JNI/Win32/JNI/JVM/... sandwich was failing when complexity of ported code got high enough. Code ran differently in Eclipse with debugging, or in Eclipse without debugging, or without Eclipse. Debugging this nightmare was almost impossible because of different random failures all over the place.

Is it possible?

There are some Java Source to C++ source converters, from Tangible Software Solutions or an online converter from Kalki Code, but online tool is not usable for production usage, and Tangible converter specifically warns about mismatches in Java and C++ code behavior.

There was a discussion on Stack Overflow where most answers were 'You just can't do that, the languages are too different'. However, all those answers missed a simple point: Java is a Cat 3 language that runs on Cat 2 JVM runtime. Each fragment of Java bytecode executes on JVM and some fragment of JVM implements that behavior. There are also parts of JVM that execute ion background and provide the rest of functionality, such as garbage collector and threading, for example. Therefore, since all Cat 3 language runtime behavior is provided by Cat 2 software, it is possible to write Cat 2 software that perfectly matches behavior of Cat 3 application. The complexity of such Cat 2 software could be easily comparable to implementation of JVM, but nonetheless it is possible.

My needs are quite specific so developing a new tool is not a bad way to go. I don't need entire JVM, no specific garbage collection if there are C++ features to implement smart pointers that take care of deallocation of objects through reference counters and whatnot. Garbage collection is not necessary if code is written in a way that when reference goes out of scope then deallocation can be immediately executed if there are no other references to same object. If each object has a reference count, garbage collection would work better only in the case if we have a data structure with circular references. In this case reference counted deallocator would leave some objects allocated forever, leaking memory. MFC memory debug code would point that out eventually, and we would take care of it then.

How?

There are many possible paths that start with Java source and end with compiled native executable.

  1. We can write a Java compiler that generates straight native code. Each Java bytecode would be replaced by somewhat equivalent native code. Debugging this output would be the most painful part of development.
  2. We can write a Java compiler that has assembly code for output. Each Java bytecode would be replaced by a list of assembly instructions. Debugging this output would be easier, but not by much.
  3. We can write a Java compiler that produces C++ code for output. Each Java bytecode would be replaced by some C++ code. Debugging of output code would be easiest, but debugging the code generator part of the compiler would be still quite complex.
  4. We can compile the Java code using a regular compiler, and translate bytecodes into C++ code. This would be easiest overall, even if it involves most steps - compile Java, convert bytecode, compile C++. There are enough Java compilers to choose from, the output is trivial, and in most cases generating C++ code will be quite easy, based on one single instruction of the bytecode at a time. Debugging will also be easiest, each line of C++ will be equivalent to one or more bytecodes, in most cases one C++ line of code will match one Java line of code.

I will pick the last option as being easiest, and will see where it goes. Instruction implementation page will combine all instruction-specific design notes.

Significant differences between Java and C++

There are few significant differences between the two that need to be taken care of. Some differences are specific to bytecode instructions, some are caused by design decisions in the JVM.

Type erasure, instruction NEW

Template system in Java is much simpler compared to C++. Generic arguments must be classes derived from Object or some specific class. A class with generic parameters can always expect that interface of the generic parameter is same as interface of the base class used to declare the generic parameter.

ArrayList always has generics that are same as Object, for example. Therefore, the concrete type of generic is not needed for correct implementation of ArrayList if the parameter is a String, since the interface ArrayList cares about is same as Object. ArrayList does not need to do anything differently just because it stores strings and not plain objects. However, when declaring a field of ArrayList type, in C++ we need to know the exact type of the template parameter.

Declarations are easy if the field has a Signature attribute. Signature attribute contains all generic parameters for the field.

Allocating a new object is more difficult. Bytecode executes NEW instruction and the type in the instruction is without generic parameters. JVM just does not need to know what types are erased. C++ code needs to create an ArrayList equivalent object with all the template parameters, but the parameter list will only be known when the result of the allocation is being stored into a field with that Signature attribute.

Garbage collection

Java has garbage collection because there is no explicit way to deallocate memory. In C++ we will use pointer wrapper objects that will be automatically deallocated at the end of scope. Destructor will decrement the usage count of an object, and will delete the object if usage counter becomes 0.

The allocation part for the pointer will also be less trivial. When pointer is created, it will be null. Each time when pointer is assigned, instead of direct assignment there would be a function that checks old pointer value for null, if non-null it will decrement usage counter of old object, then pointer will point to new object, and new object usage counter will be incremented to reflect that. The devil is in details but all this should be doable. May be even C++ operator overloading will make this easier to follow, but a dedicated function is far easier to debug.

Stack-allocated objects

C++ has capability to allocate objects on stack, and MFC uses these objects quite extensively. Java always uses dynamically allocated objects because the garbage collector cannot deal with objects on stack. To make C++ output code use stack objects to match MFC usage pattern, we can create an annotation that we add to the object reference in Java, and recompiler will generate different code when this object is being created.

Deallocating this object at the end of scope will also be different, but the presence of annotation should tell us that there are no pointers and usage counters involved.

Passing objects by value

This is a somewhat difficult one to decide how to implement.

The primary requirement of recompiler is that C++ code must be semantically same as Java code, so if Java code is executed we get same output and behavior as if we execute the C++ output code.

Java does not support passing objects by value. The Java code would have to use an intermediate, possibly generic, object that contains the copy of object to be passed as value. The called function will take this wrapped object and will unwrap it internally for the called function. The wrapper is necessary so that for each pass by value new object is created instead of old object being passed around multiple times.

However, this wrapping and unwrapping must be undone when C++ code is being generated. We'll deal with this when the time is right.

Runtime library objects

Many objects in java.lang and other java.x namespaces are performing special functions that are not easily convertible from raw Java to C++. These objects, Thread and String for example, will be manually implemented and not converted from Java source code.
