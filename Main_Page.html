Boring Web Site

This is a web site for my pet projects. The list is derived from Challenging projects every programmer should try and More challenging projects every programmer should try with some tweaks. Most people would probably find most of it just plain boring, but this stuff is interesting enough for me to try.

So what is boring?

  • MediaWiki Converter: The converter from MediaWiki pages to a static simplified web site that is suitable for hosting on GitHub pages. The conversion is only in one way for now, from MediaWiki to GitHub.
  • Secure Git Client: The idea is to have a secure Git repository on a public server such as GitHub. GitHub allows having free private repos but their content, while being invisible to the public, is still visible to the GitHub and Microsoft. Secure Git Client encrypts the repository before it gets pushed to the remote host. An example of such repository is at Secure Repo Test.
  • Recursive Descent Parser Generator: All compilers use something to parse the source code. Most parser generators generate state machine based parsers that are unfit for human consumption. The parser I'm toying with uses recursive descent, so the structure of the parser is much more similar to the structure of the grammar.
  • Java Recompiler: The main inconvenience of Java app is that it needs JVM installed and the command line is filled with more of the irrelevant stuff to execute a JAR file, compared to a simple EXE that does not need any of that overhead. Recompiler would take Java bytecodes and convert them into compilation-ready C++.
  • Widget library: Something along the lines of MFC, SWT, AWT, WX, QT or similar. Definitely in Java, converted to C++ with the bytecode converter above. Why, if some of these libraries are already in C++? Java is much more ergonomic as a language, but the output of C++ is better to execute on any Intel x64 or ARM AArch64 platform.
  • Text editor: Something that is closer to Eclipse than to Notepad. It will be again in Java as the primary language, with conversion to C++ to compile into native code. It will use the widget library from above.
  • A boring programming language. Why? Because Java/C# are too far away from actual hardware and C/C++ are too inconvenient to deal with, too much duplication of information between headers and sources, too many features missing that are taken for granted in Java and C#, and I'm not talking about garbage collection.
  • A boring compiler for the above: Language without a compiler is rather pointless, even if it is at least mildly entertaining.
  • Small operating systems: In the past I developed a RTOS for 32-bit processors (ColdFire, ARM7, ARM9). It was lots of fun then, and would be a lot more fun now for 64-bit processors such as Intel x64 and ARM v8.
  • Programming Languages: classification, etc.
