Why New Boring Programming Language?

There are tons of exciting programming languages out there. Some are even declared 'most loved' by developers that care enough to declare that they love them. There are languages that developers use but may dislike because these languages do not have enough features to do something simple. There are also languages that developers hate.

Rust may be 'most loved' but this still has not caused a 'Rust revolution'. There was a 'Swift revolution' when Apple decided to dump Objective C. There was a 'Kotlin revolution' when Google decided to ditch the Oracle sandbox and use a different 'Pascalized' version of Java to avoid lawsuits. Microsoft is trying out Rust now (2021), there are pages on how to start app development with Rust, but they seem to be far away from using Rust in any significant amount in the Windows OS itself.

There are many web pages that compare programming languages in various ways. There is of course a page on Wikipedia that has a huge matrix of checkboxes for criteria that those programming languages satisfy. There are other examples which pick some languages and give a brief description of each so that you can come up with comparison by yourself.

These comparison pages usually miss the point. When Java was a new thing, there were countless web pages talking about how Java should be more like C++, or how C++ should evolve to be more like Java. All languages are criticized many times, once when that language is released, and if the language is any relevant it will be criticized when other languages are released. Web is full of these pages.

To add to this mess, I'll toy with a simple boring language of my own. If there will ever be a web page criticizing this programming language, I'll take it as s compliment.

Current state of programming languages

To figure out what programming languages are or are not, I'll divide relevant languages into categories. Languages that are not included in this categorization are either not too general purpose (SQL for being a very specific language in a very specific domain of databases, for example), or just being functional. Functional languages are always shown as a means to calculate a factorial of something or how to sort something. Noone ever developed an operating system implementation of which absolutely required to calculate a square root of each element of a sorted array filled with factorials of last decimal digit of N first Fibonacci numbers.

Category 1: Assemblers

Assemblers are most powerful in terms what they can do and least convenient in terms of how much work they need to get things done. Each computing architecture has at least one variant of assembler. Intel used to have two, one with ATT and another with GNU syntax, or something like that.

Instruction Set Support

Assemblers support all available instructions in the target computing architecture. Assemblers can produce any instructions in any sequence, even if the resulting program is meaningless.

If the architecture is newer than assembler translator for the architecture then newer instructions can be implemented as a stream of bytes that matches the new instruction. These streams of bytes can also produce sequences that are not valid instructions on the architecture.

Typing

Assemblers are typeless. While data has no specific type, it has size. Size is determined by the instruction, each instruction operates on data of specific sizes.

Assembler program can fetch two pieces of data from 64-bit locations in memory and store them into floating point registers or into general purpose registers, and then add them together, as IEEE-754 format floating point values if data is in floating point registers, or as integers or as a pointer and integer. However, there's nothing to stop the program from fetching a pointer and a double into 64-bit registers and then adding, or-ing or xor-ing them together. The result is often meaningless but the program is more than happy to calculate the result without anyone complaining about that nonsense during compilation or execution.

Memory Management

Memory management in assembly apps is all manual. There are conventions for determining the total available amount of memory, and conventions to subdivide that memory into smaller pieces for the data.

Pure assembler apps usually run on 'bare metal'. In embedded systems this is the code that sets up and initializes the environment for languages like C or C++. Similarly, in PCs assembler is used to implement first code in BIOS that allows the rest of BIOS to be written in C. This initial code determines the total amount of available memory. This is done through conventions such as Serial Presence Detect that is used to determine sizes for each present memory module. In microcontroller apps the amount of memory is usually known, it depends on the microcontroller that is being used. App may read microprocessor device identification register and use that data to determine the amount of memory through a look table.

Subdividing that memory is based on another set of conventions. The initial amount of memory that is available for dynamic allocation is usually framed as a single block with its address and length and possibly some other housekeeping data stored in a memory control block. To allocate the data the app code creates another memory control block some distance away in the middle of current memory block and modify the first memory control block to reflect the change. To deallocate a memory block, the memory control block of that memory is marked as available, and the memory can be combined with free blocks of memory that are located before and/or after that new free block.

Category 2: C, C++, Pascal, Oberon, Modula

Category 2 languages were developed during 1970s and 1980s, C in 1972, C++ in 1985, Pascal in 1970, Oberon in 1987, Modula in 1975, Objective-C in 1984. Pascal, Oberon and Modula had few successors. The longest-lived Pascal variant is Delphi, it is probably still in top 20 of popularity indices. These languages were so adequate for their purpose at the time that there was no significant development in this category since then. The languages evolved or died, the ones that survived got more features, but no new significant languages in this category were introduced.

Instruction Set Support

Cat 2 languages are usually translated into native instruction set of the architecture that the app is meant to be executed on. Instruction set of compiled app may be different from instruction set of the compiler, in this case the compilation process is more specifically called cross-compilation. This does not change the concept, the app source code is translated into a sequence of native instructions for the target computing architecture.

Cat 2 languages produce specific instruction sequences that include certain instructions in certain order. Not all instructions can be represented by the language itself. There are tricks external to Cat 2 languages to generate these extra instructions. There are intrinsic functions that look like native functions from the language perspective, but compilers know how to handle these functions differently. There is a concept of 'inline assembly' that allows writing assembly code within the Cat 2 program. Strictly speaking, this inline assembly is not Cat 2 program but a fragment of Cat 1 language code that is placed inside a Cat 2 program code.

Typing

All entities in Cat 2 programs have a type. Type specifies which values can be stored in a variable of that type. Entities are created by specifying a type for that entity. Types can be primitive types that are built into the language, and types can be user-defined and constructed of other types, including primitive and user-defined types that are defined before that new user-defined type can be created.

Typing in Cat 2 languages is usually weak. Entity has a type, but language allows treating any entity of one type as an entity of any other type, and these two types don't even have to be related. All these type casts must be instantiated manually, so there is some type safety, but, when necessary, these type casts can be inserted to accomplish something that would be otherwise impossible if the typing rules were to be followed strictly.

The best example of where typing must be defeated is the memory allocation and deallocation code. The initial memory layout is a big sequence of bytes with a memory control block in front of it. When memory is allocated, new memory block is created in the middle of that sequence of bytes, and the address of the beginning of first sequence of bytes is returned as an address of allocated memory block. In this process first type cast is when a Memory Control Block entity is created in the middle of the byte sequence, and second type cast happens when an address of byte sequence is returned to the user app as a pointer to newly allocated memory of their desired type. Without type casts Cat 2 language typing system would not let the code compile.

Memory Management

Memory management in Cat 2 languages is all manual. App developer decides when the memory must be allocated and when it can be deallocated.

Manual allocation can lead to memory inconsistencies. In manual memory allocation cycle the sequence of events is following:

  1. App invokes the API to allocate the memory.
  2. A pointer to allocated memory is returned to the app.
  3. A pointer is passed when app invokes API to deallocate memory.
  4. The original pointer and all its copies are still available to the app.
At step 2 the pointer to the memory is available to the app. App can pass the pointer around and unlimited copies of that pointer can be made. When in step 3 the pointer is passed to deallocation API, the pointer value is not destroyed, any copies of that pointer are not destroyed either. This leads to 'use after free' or 'free after free' bugs. These bugs are inherent to manual allocation because the pointer from step 2 is necessary for deallocation but it exists in countless copies even after the deallocation in step 4.

There are third party libraries for automatic memory management using garbage collection, but from the language perspective they are just a set of APIs for allocating memory that is tracked by that third party library and deallocation happens from within that library. The library still uses manual memory deallocation facilioties of the language, but user app is developed without invoking the deallocation API explicitly. In this case, step 1 is slightly modified to conform to the third party library API, step 2 is not changed, and steps 3 and 4 sre not available because the third party library executes step 3 only after all pointers from step 4 disappear by going out of their scopes.

Interoperability

Cat 2 languages are fully interoperable with Cat 1 languages if the Cat 1 language is the one that matches the target execution environment. Cat 1 language can produce any sequence of instructions, therefore it can produce a sequence of instructions that any Cat 2 language would produce.

Cat 2 compiled code can be invoked from Cat 1 code using exactly same instruction sequence as Cat 2 would use to invoke it. Cat 1 code can be invoked from Cat 2 code if Cat 1 code follows ABI that Cat 2 languages use for subroutine/method/function entry and exit code, parameter passing and for register allocation within that code fragment.
